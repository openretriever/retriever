General-purpose robot agents rarely fit in one clean loop. Cameras stream, robot state updates faster than policies, VLM/VLA calls have variable latency, planners can block, operators intervene, and replay has to explain exactly what happened. Retriever puts timing and handoff decisions into the graph instead of scattering them across callbacks, queues, and sleeps.

## The four objects

<div class="card-grid">
  <a class="info-card" href="/concepts/flow/"><strong>Flow</strong><span>A typed Python class with local state and a synchronous `step(...)` method.</span></a>
  <a class="info-card" href="/concepts/time-and-sync/"><strong>Clock</strong><span>Each Flow declares when it runs; there is no global robot timestep.</span></a>
  <a class="info-card" href="/concepts/time-and-sync/"><strong>Sync policy</strong><span>Each edge declares how upstream event history becomes one aligned input.</span></a>
  <a class="info-card" href="/concepts/runtime/"><strong>Pipeline</strong><span>The graph that validates, visualizes, steps, replays, and runs.</span></a>
</div>

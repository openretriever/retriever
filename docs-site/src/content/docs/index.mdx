<p class="mental-model"><strong>Mental model:</strong> PyTorch composes <code>nn.Module</code>s that run immediately; Retriever composes typed <code>Flow</code>s that run in explicit time. Each Flow owns local state, wakes on its own clock, consumes one synchronized input record, and can be stepped locally before it runs on a backend.</p>

## Quick Start

Run the source-checkout demo first. It proves the runtime graph without camera permission, GUI windows, Rerun, or robot hardware.

```bash
git clone https://github.com/openretriever/retriever.git
cd retriever
retriever install --bootstrap-pixi
retriever run webcam-mock
```

<details class="explain-details">
  <summary>What does this do?</summary>
  <div>
    <p><code>retriever install --bootstrap-pixi</code> prepares the source checkout. <code>retriever run webcam-mock</code> runs a small graph: synthetic camera frames feed a color detector, then detections are printed to stdout. Use the webcam/Rerun path only after this smoke works.</p>
  </div>
</details>

Use the source checkout for demos today: it includes examples, tutorial assets, graph rendering, and optional visualization. The runtime-only distribution name is `retriever-core`, and the Python import remains `retriever`; see [Install](/getting-started/install/) for the package/source split.

## Why Retriever For Robotics

<div class="check-list">
  <p><strong>Compositional Flow contracts.</strong> Perception, memory, planning, skills, monitors, and controllers stay as reusable typed modules instead of callback glue.</p>
  <p><strong>Explicit multi-rate time.</strong> Cameras, model calls, skills, and controllers do not need to pretend they share one global timestep.</p>
  <p><strong>Deterministic input handoff.</strong> Each edge declares how upstream event history becomes the one input record consumed by <code>step(...)</code>.</p>
  <p><strong>Debuggable before deployment.</strong> Render the graph, step locally, use stdout/Rerun, record consumed inputs, and replay the same run.</p>
  <p><strong>Backend mapping when ready.</strong> Keep the graph semantics stable while moving from in-process stepping to multiprocessing or Dora-backed execution.</p>
</div>

## Common Patterns

| Need | Start with |
| --- | --- |
| First reliable smoke | `retriever run webcam-mock` |
| Live visual path | `retriever run webcam` |
| Smallest Flow mental model | `retriever run basic-flow` |
| Graph inspection | `retriever run graph` |
| Record and replay | `retriever run record` then `retriever run replay` |
| Hub ref inspection | `retriever hub parse openretriever/hello-world:HelloFlow` then `retriever hub inspect openretriever/hello-world --json` |
| Applied robot examples | GoldenRetriever first proof, then the example catalog |

For checkpointable debugging, use the Python stepping API: `Pipeline.step(...)` or the top-level `retriever.step(...)` helper on the active pipeline. Saved IR/HTML is for inspection and reproducibility; Python remains the executable graph source.
